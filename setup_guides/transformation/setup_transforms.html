设置转换

在本指南中，我们将研究Nav2所需的必要转换。这些变换允许Nav2通过将信息转换为使用的坐标框架来解释来自各种来源的信息，例如传感器和里程计。下面是机器人的全变换树的样子，但我们将从更简单的东西开始。

在本教程中，我们将首先简要介绍ROS中的转换。其次，我们将研究TF2静态发布者的简单命令行演示，以查看其运行情况。最后，我们将概述为使Nav2发挥作用而需要发布的必要转换。

转换介绍

注解

本指南的这一部分改编自ROS (1) 导航文档中的 Setting Up You Robot using tf 教程。

许多ROS包要求使用TF2 ROS包发布机器人的变换树。变换树定义了不同坐标系之间的关系，包括平移、旋转和相对运动。为了使这更具体，让我们应用一个简单机器人的例子，它有一个移动底座，上面安装了一个激光传感器。

这个机器人有两个定义坐标系: 对应中心点移动底座机器人，中心点激光安装的底座。我们呼叫坐标框架附在移动底座 base_link 我们会呼叫坐标框架附在激光 base_laser 。注意将讨论更多关于命名和公约这些坐标系节。

此时，让's assume that we have some data from the laser in the form of distance measurements from the laser'的中心点。换句话说，我们在 base_laser 坐标系中有一些数据。

现在，假设我们想要获取这些数据并使用它来帮助移动基地避免世界上的障碍。为了成功地做到这一点，我们需要一种方法将我们收到的激光扫描从 base_laser 框架转变为 base_link 框架。本质上，我们需要定义 base_laser 和 base_link 坐标框架之间的关系。

在定义这种关系，假设唯一数据是激光安装10厘米和20厘米以上的中心点移动底座。这给我们一个平移偏移关联 base_link 框架 base_laser 框架。特别是，我们知道获取数据从 base_link 框架 base_laser 框架，我们必须应用翻译 (x: 0.1m，y: 0.0m，z: 0.2m)，横向，获取数据从 base_laser 框架 base_link 框架，我们必须利用相反的翻译 (x:-0.1m，y: 0.0m，z: -0.20m)。

我们可以选择管理这个关系，意义存储和应用适当翻译帧之间必要时，但这成为真正痛苦的坐标系增加。幸运的是，我们不必做这工作自己。相反，我们将定义关系 base_link 和 base_laser 一旦使用TF2让管理转型两坐标系我们。这是特别有用当使用非静态转换，如一套帧相对运动，像机器人底座地图框架。

定义和存储关系 base_link 和 base_laser 帧使用TF2，我们需要添加到变换树。概念，每个节点变换树对应坐标系，每个边缘对应变换需要应用从当前节点其子。TF2使用树结构来保证只有一个遍历链接任何两个坐标系一起，假定所有边树定向到子节点。

创建转换树简单的例子，我们将创建两个节点: 一个 base_link 坐标系之一的 base_laser 坐标系。创建边缘之间，我们需要决定哪个节点将父母将孩子。记住这个区别很重要，因为TF2假定所有变换从父儿童。

让我们选择 base_link 坐标系父因为当其他/传感器添加到机器人，它将最适合他们有关 base_laser 框架遍历通过 base_link 框架。这意味着变换与边缘连接 base_link 和 base_laser 应 (x: 0.1m，y: 0.0m，z: 0.2m)。

通过设置这个变换树，将在 base_laser 框架中接收到的激光扫描转换为 base_link 框架就像调用TF2库一样简单。我们的机器人现在可以使用这些信息来推理 base_link 框架中的激光扫描，并安全地围绕其环境中的障碍物进行计划。

静态转换发布者演示

警告

如果您是ROS 2的新手或还没有工作环境，请花一些时间使用官方 ROS 2 Installation Documentation _ 中的资源正确设置您的机器

现在，让我们尝试使用tf2提供的static_transform_publisher变换工具发布一个非常简单的变换。我们将发布从链接 base_link 到链接 base_laser 的转换，翻译为 (x: 0.1m，y: 0.0m，z: 0.2m)。请注意，我们将从本教程前面的图表构建转换。

打开命令行并执行以下命令:

ros2 run tf2_ros static_transform_publisher 0.1 0 0.2 0 0 0 base_link base_laser

,我们现在成功出版我们 base_link 转 base_laser 变换TF2.让我们现在检查它是否正常工作通过 tf2_echo 。开辟单独命令行窗口执行以下:

ros2 run tf2_ros tf2_echo base_link base_laser

你应该能够观察到一个重复的输出simiar到下面的那个。

At time 0.0
- Translation: [0.100, 0.000, 0.200]
- Rotation: in Quaternion [0.000, 0.000, 0.000, 1.000]

这是它短演示我们可以成功发布变换从 base_link 转 base_laser 使用TF2图书馆。注意，我们不建议使用上述演示发布转换实际机器人项目，只是一个快速演示看到TF2行动。用实际机器人系统，我们将创建URDF文件嵌入信息的机器人使用robot_state_publisher而不是static_transform_publisher。有更为简便方式去这将讨论 设置URDF 教程。

参见

如果你想了解更多关于TF2的信息，以及如何创建自己的转型出版商，请前往官方的 TF2 Documentation _

我们强烈建议您查看两个重要的ROS代表。这些文件详细说明了ROS社区制定的一些标准，以确保不同包之间的正常运行。Nav2也遵守这些标准和惯例。

  1. REP 105 - Coordinate Frames for Mobile Platforms

  2. REP 103 - Standard Units of Measure and Coordinate Conventions

为了快速总结REP 105，本文档指定了ROS中使用的不同坐标帧的命名约定和语义。本教程感兴趣的是 base_link 、 odom 和 map 坐标框架。 base_link 是连接到机器人固定位置的坐标框架，通常在其主底盘和旋转中心。 odom 坐标框架是相对于机器人起始位置的固定框架，主要用于局部一致的距离表示。最后， map 坐标框架是一个世界固定框架，用于距离的全局一致表示。

REP 103，另一方面，探讨标准单位和其他有关公约保持集成问题不同ROS套餐最低。基本概述帧定义右手规则，Z和X，和单位应标准SI单位。

现在我们继续一些细节的Navigation2包的功能正确。Nav2需要以下转换公布ROS:

  1. map => odom

  2. odom => base_link

  3. 这个``base_link`` => base_laser (传感器底座)

注解

的 base_laser 坐标系不带REP 105标准。对于本指南，我们将使用这个名称指坐标系激光传感器的机器人平台。如果有多个传感器底座 (例如camera_link，base_laser2，lidar_link等)，然后转换回 base_link 每个需要。

第一个变换 map => odom 通常由处理定位和映射的不同ROS包提供，例如AMCL。这个转换更新在使用中，所以我们不使用't set static values for this in our robot'的TF树。关于如何设置它的更多细节可能非常复杂，所以我们强烈建议您看看您在平台上使用的映射或本地化包的文档。所有符合ROS标准的SLAM和本地化包将在发布时自动为您提供此转换。

的 odom => base_link 通常由我们里程计系统传感器如车轮编码器。这是通常计算通过传感器融合里程计传感器 (IMU，车轮编码器，VIO等) 使用 robot_localization 包。

所有其他静态定义变换 (e。g. base_link => base_laser ， base_link => wheels ， wheels => IMU 等) 是我们将谈论本指南的其余部分。这种转变树使用Nav2正确联系信息传感器或其他框架兴趣的机器人。改造两坐标系通常提供Nav2通过机器人状态出版商和通用机器人描述符文件 (URDF)。如果有更多传感器坐标系上平台，然后变换树 base_link 每个传感器坐标系需要发布。

结论

在本教程中，我们讨论了变换的概念以及它们如何在Nav2中使用。

在最后一节中，我们也探索了使用TF2的static_transform_publisher来发布我们的转换。您可以使用它来设置Nav2的转换，但这通常不是最好的方法。在大多数机器人项目中，我们使用机器人状态出版商，因为它更容易使用，并且随着我们的机器人变得更加复杂而扩展。我们将讨论机器人状态出版商URDF，以及如何在下一个关于 设置URDF 的教程中设置它。
